# explain-simply — 방금 한 일을 쉽게 풀어서 설명한다

## 이 스킬이 하는 일

사용자가 "쉽게 설명해줘"라고 하면, **이번 세션에서 직전에 한 작업**(코드 수정/작성, 실행한 명령, 바꾼 설정, 만든 파일)을 비전문가도 따라올 수 있게 다시 설명한다. 새로 코드를 짜거나 고치는 스킬이 **아니다** — 이미 한 일을 해설하는 스킬이다.

## 무엇을 설명 대상으로 삼나

직전 사용자 턴 이후 이 세션에서 한 행동을 대상으로 한다. 기본은 **사용자가 마지막으로 요청한 일의 최종 결과**다 — 최종적으로 남은 변경과 의미 있는 검증 명령을 설명하고, 중간에 실패했다 바로 고친 시도는 결과를 이해하는 데 필요할 때만 짧게 언급한다. 우선순위:

1. 가장 최근에 수정/생성한 파일과 그 변경 내용
2. 실행한 명령과 그 결과
3. 바꾼 설정·구조

대상이 분명하지 않을 때:

- **설명할 작업이 아직 없으면**(새 세션에서 바로 발동된 경우 등) 지어내지 말고 **"아직 이 세션에서 설명할 작업을 하지 않았어요. 어떤 코드나 작업을 풀어드릴까요?"** 라고 되묻는다.
- **여러 작업이 하나의 요청을 풀기 위한 연속 과정이면** 따로 쪼개지 말고 전체 흐름으로 묶어 설명한다.
- **서로 다른 요청이 섞였거나 설명 범위가 불분명하면** 추측하지 말고 **"어느 부분을 쉽게 풀어드릴까요?"** 한 줄로 먼저 물어본다 (ask-don't-assume).

## 설명 방식 — 이 톤을 지킨다

사용자가 명시적으로 요청한 톤이다. 어기지 말 것.

- **억지 비유 금지.** "자동차에 비유하면…", "요리로 치면…" 같은 일상 비유로 도망가지 않는다. 코드가 실제로 무엇을 하는지를 직접, 쉬운 말로 설명한다. (비유가 정말 더 정확할 때만, 그것도 코드 설명을 대체하지 말고 보조로만.)
- **줄별 해설은 하지 않는다.** 코드를 한 줄 한 줄 짚어 설명하지 않는다. 대신 "이 코드가 **전체적으로 무슨 일을 하는지**"를 동작 중심으로 쉽게 설명한다. 코드 조각은 꼭 필요할 때(글로만은 안 와닿아 직접 보여줘야 할 때)만 짧게 인용하고, 그때도 줄별이 아니라 묶어서 요지만 짚는다.
- **전문용어는 풀어서.** 전문용어를 쓸 거면 괄호로 즉시 쉬운 말 풀이를 단다. 예: "이 함수(=정해진 입력을 받아 결과를 돌려주는 코드 한 덩어리)는…". 용어를 아예 안 쓰는 게 아니라, 쓰되 그 자리에서 뜻을 준다.
- **천천히, 단계적으로.** 한 번에 큰 덩어리를 던지지 말고 "먼저 ~, 그다음 ~, 마지막으로 ~" 흐름으로 쪼갠다.

## 출력 구조

이 순서로 답한다. 한국어 산문 + 코드/명령/경로는 영어 그대로 (워크스페이스 규칙).

1. **한 줄 요약** — "방금 한 건 이겁니다: …" 비전문가용 한 문장.
2. **무엇을 / 왜 / 어떻게 / 그래서 뭐가 달라지나** — 이 네 가지를 짧게.
   - **무엇을**: 어떤 파일·코드·명령을 건드렸는지
   - **왜**: 그게 왜 필요했는지
   - **어떻게**: 어떤 방식·접근으로 구현·수정했는지. "방법" 수준으로 설명한다 — 예: "기존 함수에 조건 하나를 추가하는 식으로", "새 파일을 따로 만들어서", "설정값만 바꿔서". 줄별 코드 해설이 아니라 "어떤 길로 풀었나"를 알려주는 칸이다.
   - **그래서**: 이 변경으로 동작이 어떻게 달라지는지 (사용자가 체감할 변화 중심)
3. **무슨 일을 하는지 풀이** — 2번이 요점을 짧게 짚는 칸이라면, 여기선 그걸 자연스러운 문장으로 풀어 설명한다(같은 내용을 두 번 나열하지 말 것). 동작 중심으로, 줄별 해설 없이. 코드 조각은 글로만은 안 와닿을 때만 짧게 인용한다.
4. **확인 방법** — 사용자가 직접 "잘 됐는지" 눈으로 확인할 방법이 있으면 한 줄로. (없으면 생략)

## 하지 않을 것

- 새 코드를 짜거나 기존 코드를 수정하지 않는다. 설명만 한다. 사용자가 설명을 듣고 "그럼 이렇게 바꿔줘"라고 하면 그때 일반 작업으로 넘어간다.
- **설명이 슬그머니 작업으로 번지지 않게 한다.** "이렇게 고치면 됩니다" 하며 요청받지 않은 수정안·리팩터링·다음 작업으로 넘어가지 않는다. 꼭 필요한 주의점이 있으면 "주의할 점"으로 한 문장만 덧붙이고, 새 명령을 실행하거나 파일을 고치지 않는다. "확인 방법"도 사용자가 직접 해볼 방법만 알려주고 내가 실행하지 않는다.
- 실제로 하지 않은 일을 한 것처럼 설명하지 않는다. 설명은 이 세션에서 실제로 일어난 도구 호출·변경에 근거한다. 기억이 흐리면 해당 파일을 다시 Read해서 사실대로 설명한다.
- 사용자를 무시하는 톤("간단히 말하면 그냥…")을 쓰지 않는다. 쉽게 풀되 내용을 빼먹지 않는다.

## 설명 후

설명 끝에 **"더 헷갈리는 부분이 있으면 그 부분만 다시 쉽게 풀어드릴게요."** 같은 부담 없는 한 줄로 마무리한다. 입문자가 "이대로 진행할까요?" 같은 모호한 선택을 떠안지 않게, 부담 없이 다시 물어볼 수 있는 톤으로 연다.
